# Audit qualité — Cœur métier Diapason

**Date :** 2026-04-26
**Périmètre :** parsing XLIFF, modèle objet, vérifications de cohérence cross-locale, formatage canonique. CLI/Console et chargement de configuration sont **hors périmètre**.
**Axes :** lisibilité & nommage · modularité & couplage · robustesse & types.

L'audit confronte le code de Diapason aux patterns observés dans des codebases PHP de référence (Symfony 8.1, PHPUnit `main`, PHP-CS-Fixer `master`, PHPStan 2.1.x, Crell/Serde, Rector, API Platform). Toutes les références externes sont des liens GitHub directs vers `main` ou la branche stable courante.

---

## 1. Vue d'ensemble — ce qui est déjà solide

À conserver, et à mettre en avant pour les contributeurs :

- `declare(strict_types=1);` **dans tous les fichiers du cœur**.
- `final readonly class` partout où c'est sémantiquement juste (`XliffFile`, `Unit`, `Group`, `Segment`, `Issue`, `DomainBundle`, `ProjectReport`, `FormatPreview`, `XliffParser`, `Checker`, `DiffReporter`).
- Constructor property promotion + named arguments à l'appel — alignés avec Symfony récent et Crell.
- Backed enums (`Severity: string`, `IndentStyle: string`) — déjà correct pour la sérialisation JSON.
- Constantes de classe typées PHP 8.3 (`private const string XLIFF_NS = …` dans [src/Parser/XliffParser.php:18](../src/Parser/XliffParser.php#L18)) — plus moderne que ce qu'on voit chez Symfony Translation 8.1.
- Découpage `Model/` · `Parser/` · `Check/` · `Formatter/` · `Reporter/` · `Exception/` clair, avec interfaces `PerFileCheckInterface` / `CrossLocaleCheckInterface` qui découplent l'orchestration du contenu des règles.
- Tests organisés en miroir de `src/`, avec **21 fixtures** XLIFF couvrant les scénarios métier (valid, malformed, wrong namespace, missing group, source mismatch, etc.).

Le socle est sain. Les recommandations qui suivent visent à **densifier la signature contributeur** : moins de méthodes monolithiques, conteneurs typés au lieu de tableaux nus, fronts d'erreur explicites.

---

## 2. Méthode

Deux explorations parallèles ont alimenté ce rapport :

1. **Cartographie interne** — inventaire exhaustif du cœur métier, classe par classe (responsabilité, modificateurs, dépendances entrantes/sortantes, longueur, points chauds par axe).
2. **Recherche externe** — lecture directe des fichiers de référence dans :
   - `symfony/symfony` (Translation/XliffUtils, Translation/Loader, Translation/Dumper, Validator/ConstraintViolationList, DomCrawler/Crawler, Console)
   - `sebastianbergmann/phpunit` (Util/Xml/Loader, TextUI/Configuration/Xml/Loader, Framework/TestStatus)
   - `PHP-CS-Fixer/PHP-CS-Fixer` (FixerInterface, AbstractFixer, FixerDefinition, Error, AbstractFixerTestCase, fixtures `.test`)
   - `phpstan/phpstan-src` (Analyser/Error, ShouldNotHappenException)
   - `Crell/Serde` (Serde, SerdeException, UnsupportedFormat, Direction, Attributes/ClassSettings)
   - `rectorphp/rector-src` (PhpParser/Node/BetterNodeFinder)
   - `api-platform/core` (Metadata/ApiResource)

Chaque pattern listé ci-dessous est sourcé par un fichier+ligne vérifiable.

## 3. Patterns externes retenus

### Lisibilité & nommage

- **L1 — Une classe = un rôle, courte.** PHPUnit dédouble son XML loading : un `Util\Xml\Loader` (~95 lignes, charge un `DOMDocument`, gère `libxml_use_internal_errors`) ; un `TextUI\Configuration\Xml\Loader` qui ne fait *que* l'interprétation métier. Réfs : [Util/Xml/Loader.php](https://github.com/sebastianbergmann/phpunit/blob/main/src/Util/Xml/Loader.php), [TextUI/Configuration/Xml/Loader.php](https://github.com/sebastianbergmann/phpunit/blob/main/src/TextUI/Configuration/Xml/Loader.php).
- **L2 — Factories statiques nommées au lieu de constructeurs ambigus.** PHPUnit `TestStatus::success()`, `::failure($msg)`, `::from(int)` ; Crell `UnsupportedFormat::create($format, Direction::Serialize)` qui *construit ET formule le message*. Réfs : [TestStatus.php](https://github.com/sebastianbergmann/phpunit/blob/main/src/Framework/TestStatus/TestStatus.php), [UnsupportedFormat.php](https://github.com/Crell/Serde/blob/master/src/UnsupportedFormat.php).
- **L3 — Verbes complets, prédicats `is/has`, pas de préfixes inventés.** `getNewMessages`, `validateSchema`, `isCandidate`, `isRisky`, `hasMetadataArrayInfo`. Réf : [FixerInterface.php](https://github.com/PHP-CS-Fixer/PHP-CS-Fixer/blob/master/src/Fixer/FixerInterface.php).
- **L4 — PHPDoc proportionnel à la criticité.** Zéro PHPDoc sur les enums et VO triviaux (Crell `Direction`). PHPDoc explicatif + `@throws` sur l'API publique (`XliffUtils::validateSchema`).
- **L5 — Early returns systématiques, jamais de `else` après `return`.** Tous les fixers PHP-CS-Fixer suivent ce pattern (cf. [LowercaseStaticReferenceFixer](https://github.com/PHP-CS-Fixer/PHP-CS-Fixer/blob/master/src/Fixer/Casing/LowercaseStaticReferenceFixer.php), 6 returns successifs).

### Modularité & couplage

- **M1 — Loader ≠ Parser ≠ Validator ≠ Dumper, classes séparées avec un modèle pivot.** Symfony Translation : `XliffUtils` (validation schéma + version), `XliffFileLoader` (parse vers `MessageCatalogue`), `XliffFileDumper` (sérialise depuis `MessageCatalogue`). Aucune ne se connaît au-delà du catalogue. Réfs : [Util/XliffUtils.php](https://github.com/symfony/symfony/blob/8.1/src/Symfony/Component/Translation/Util/XliffUtils.php), [Loader/XliffFileLoader.php](https://github.com/symfony/symfony/blob/8.1/src/Symfony/Component/Translation/Loader/XliffFileLoader.php), [Dumper/XliffFileDumper.php](https://github.com/symfony/symfony/blob/8.1/src/Symfony/Component/Translation/Dumper/XliffFileDumper.php).
- **M2 — `final readonly` par défaut, sauf BC explicite ou état mutable légitime.** Adoption agressive chez PHPUnit (`Loader`), Rector (`BetterNodeFinder`), Crell. Plus prudente chez Symfony et PHP-CS-Fixer (BC promise). À l'inverse, `AbstractFixer` n'est pas `readonly` car `setWhitespacesConfig` mute.
- **M3 — Interfaces uniquement aux points d'extension publics.** `FixerInterface` (les utilisateurs créent leurs fixers), `ConstraintViolationListInterface` (polymorphisme du validateur). Mais `XliffUtils`, `Crawler`, `BetterNodeFinder` sont `final` sans interface.
- **M4 — I/O confiné dans une seule classe.** PHPUnit ne fait `new DOMDocument()` que dans `Util\Xml\Loader`. Symfony `XliffFileLoader::load` délègue le chargement à `XmlUtils::loadFile()` puis travaille en mémoire. Le `try/finally` sur `libxml_use_internal_errors` est centralisé.
- **M5 — Registry de règles avec définition riche.** PHP-CS-Fixer attache un `FixerDefinition($summary, $codeSamples, $description, $riskyDescription)` à chaque fixer, ce qui débloque `php-cs-fixer describe`. Réf : [FixerDefinition.php](https://github.com/PHP-CS-Fixer/PHP-CS-Fixer/blob/master/src/FixerDefinition/FixerDefinition.php).

### Robustesse & types

- **T1 — Une exception par cas d'erreur, marker interface plate en racine.** Crell expose ~25 classes (`UnsupportedFormat`, `MissingRequiredValueWhenDeserializing`, `CircularReferenceDetected`, `TypeMismatch`…) toutes implémentant `interface SerdeException {}`. Permet `catch (SerdeException $e)` *et* `catch (UnsupportedFormat $e)` sans hiérarchie d'héritage rigide. Réfs : [SerdeException.php](https://github.com/Crell/Serde/blob/master/src/SerdeException.php), [Crell/Serde/src](https://github.com/Crell/Serde/tree/master/src).
- **T2 — Erreurs *collectées*, pas thrown.** `ConstraintViolationList` implémente `IteratorAggregate<int, ConstraintViolationInterface>`, accumule via `add()`/`addAll()`, expose `__toString()` pour le reporting. PHPStan a `Analyser\Error` riche (message, file, line, identifier, tip, metadata, fixedErrorDiff). On accumule, on ne throw que pour interrompre le pipeline. Réfs : [ConstraintViolationList.php](https://github.com/symfony/symfony/blob/8.1/src/Symfony/Component/Validator/ConstraintViolationList.php), [Analyser/Error.php](https://github.com/phpstan/phpstan-src/blob/2.1.x/src/Analyser/Error.php).
- **T3 — Generics PHPDoc systématiques.** `@implements \IteratorAggregate<int, X>`, `@var list<X>`, `@template T of Node` + `@param class-string<T>` + `@return T[]`.
- **T4 — `with*()` immutable pour les value objects modifiables.** PHPStan `Error::changeFilePath(string): self` retourne un nouveau `Error`. Pattern répété chez Crell, API Platform (`ApiResource::withOperations`).
- **T5 — `try/finally` strict autour de l'état global libxml.** PHPUnit `Util\Xml\Loader::load` :
  ```php
  $internal = libxml_use_internal_errors(true);
  try { /* … */ }
  finally { libxml_use_internal_errors($internal); }
  ```
- **T6 — Property hooks PHP 8.4 / asymmetric visibility : adoption non observée.** Symfony et PHP-CS-Fixer maintiennent une compat 8.2/8.3. Crell utilise `public readonly` (équivalent 8.1+ sans hooks). Pas de pression à adopter pour un projet 8.4.

### Spécifique XLIFF/formatter

- **X1 — DOMDocument brut, pas de wrapper.** Symfony `XliffUtils::validateSchema` fait `$dom->getElementsByTagName('xliff')`, `simplexml_import_dom($dom)` directement. → **Diapason a raison** de garder `DOMDocument` dans `XliffParser`.
- **X2 — Namespace XLIFF par constante typée + `getElementsByTagNameNS`.** Symfony : `$xml->registerXPathNamespace('xliff', 'urn:oasis:names:tc:xliff:document:2.0')`. Diapason fait déjà mieux avec `private const string XLIFF_NS` ([XliffParser.php:18](../src/Parser/XliffParser.php#L18)).
- **X3 — Modèle objet typé vs tableau associatif : choix dépendant du domaine.** Symfony Translation → tableau (`MessageCatalogue` + metadata sidecar). PHPUnit XmlConfiguration → graphe d'objets (`Configuration`, `TestSuiteCollection`, `Source`, …). **Diapason est dans le camp PHPUnit, c'est le bon choix** vu sa nature de validateur cross-fichiers.
- **X4 — Tests de formatter en fixtures `.test` à sections.** PHP-CS-Fixer stocke un fichier unique avec sections `--TEST--`/`--RULESET--`/`--EXPECT--`/`--INPUT--`. Un cas = un fichier, lisible en revue, ajout trivial. Réf : [anonymous_class_with_phpdoc.test](https://github.com/PHP-CS-Fixer/PHP-CS-Fixer/blob/master/tests/Fixtures/Integration/misc/anonymous_class_with_phpdoc.test).

## 4. Recommandations pour Diapason

> Toutes les recommandations de cet audit ont été appliquées :
> - **R1 à R4** (quick wins — `try/finally` libxml, marker interface `DiapasonException`, generics PHPDoc, génériciser les `PerFileCheck`) — commits `7a097d0`, `f785128`, `917fab7`.
> - **R5 à R8** (refactos structurants — split `XliffDomLoader`/`XliffParser`, `IssueList` itérable, `DomainVerdict` immutable, découpe de `UnitsConsistencyCheck`) — commits `a1ed9f8`, `39294ee`.
> - **R9 à R10** (long terme — `CheckDefinition` riche, fixtures `.test` sectionnelles pour le formatter) — commits `184cc65`, `cf1e88a`.

## 5. Ce que l'audit n'a *pas* trouvé

Pour cadrer l'inquiétude, voici ce que ces codebases de référence font et qui *n'apparaît pas* comme un manque chez Diapason :

- Pas de besoin de wrapper autour de `DOMDocument` — Symfony et PHPUnit travaillent en DOM brut, c'est la norme.
- Pas de besoin de property hooks PHP 8.4 / asymmetric visibility — aucune des huit codebases auditées ne les a adoptés en production. `final readonly` couvre 95 % des cas.
- Pas de `mixed` ni d'`array` non-typé dans le cœur métier — déjà discipliné.
- Pas de dépendances cycliques, pas de mutual recursion entre namespaces.

Le projet est dans un état où **les recommandations portent sur la qualité, pas sur la dette**. C'est exactement ce qu'on veut pour un outil que l'on souhaite voir adopter.
